# v0.6 后端 G3：执行策略端口收敛冻结

- 状态：已冻结，允许进入 I3 实施
- Owner：架构-小强（冻结）；全栈-小智（实施）
- 前置：G2 已通过；候选查询与健康状态域代码基线为 `a3265db`
- 本批目标：在**不改变请求执行语义**的前提下，将执行服务消费的策略判断从宽泛的 `ExecutionDependencies` bridge 收敛为显式 `ExecutionPolicyPort`；不拆服务、不改 HTTP 路由、不改数据模型。
- 本批非目标：完成全部 `ExecutionDependencies` 的消除。该 bridge 仍作为兼容层保留，其他依赖组后续另立 Gate。

## 1. 唯一 owner 与职责边界

| 领域 | 唯一 owner / 协作者 | I3 结论 |
|---|---|---|
| 候选枚举、聚合成员 skip、候选/成员健康写入 | `CandidateHealthService` → `ConfigStore` | G2 已冻结；I3 只消费，不迁移、不复制 |
| 执行策略：上游失败分类、自动路由判定、冷却时长与 WAF 提示的**纯策略结果** | 新增 `ExecutionPolicyService` | I3 唯一业务 owner |
| 路由/聚合查询与持久化 | `ConfigStore` / 既有路由查询适配 | 不迁移 |
| HTTP 请求、response 生命周期与流读取 | `UpstreamClient`、`NonStreamExecutionService`、`StreamExecutionService` | 不迁移 |
| WAF 锁与活动流计数 | `WafLockState` | 不迁移 |
| 日志、live request、debug capture | `ObservabilityService` / `DebugCapture` | 不迁移 |
| 兼容组装、既有私有辅助入口 | `ArkProxyRouter` facade | 只委托，不再拥有 I3 策略分支 |

`ExecutionPolicyService` 必须是无状态对象：不得持有 `ConfigStore`、HTTP client、锁、日志服务、请求体或 response，也不得缓存候选、模型、聚合成员或健康状态。唯一的数据 owner 仍是既有 Store/服务。

## 2. 端口、DTO 与依赖方向

### 2.1 新端口

在 `linrouter_core/contracts/execution_ports.py` 定义并使用窄 `ExecutionPolicyPort`。端口仅暴露下面四类能力；实现可保留现有同义方法的薄兼容 wrapper：

1. `classify_candidate_error(status_code, raw, error_kind="http") -> CandidateErrorClassification`
2. `is_auto_model(requested_model, group) -> bool`
3. `auto_cooldown_seconds(group) -> int`
4. `waf_blocked_suffix(classification, group) -> str` 与 `waf_blocked_hint(fallback_chain) -> str`

新增 `CandidateErrorClassification` 为具名、不可变 DTO（建议 `dataclass(frozen=True)`），字段固定为：

- `should_cooldown: bool`
- `is_request_level: bool`
- `category: str`
- `log_reason: str`
- `failure_scope: str`

执行服务可在写日志或构造 fallback chain 时读取这些字段；不得将 DTO 序列化为新的外部 HTTP 协议字段。

### 2.2 依赖规则

```text
ArkProxyRouter compatibility facade / execution services
  -> ExecutionPolicyPort
  -> ExecutionPolicyService (pure policy)
  -> existing injected predicate callbacks only

ArkProxyRouter compatibility facade / execution services
  -> CandidateHealthService -> ConfigStore
  -> UpstreamRequestPort / ProtocolAdapterPort / ConcurrencyPort / RequestObservabilityPort
```

实施约束：

- `ExecutionPolicyService`、新增 DTO、`ExecutionPolicyPort` 禁止 `import app`、`RouterHandler`、`ArkProxyRouter`、`ConfigStore`、`urllib` 或 `threading`；
- `CandidateRuntime` / 执行服务只能依赖 `ExecutionPolicyPort`，不得直接调用 facade 的 I3 策略实现；
- `ArkProxyRouter` 可保留 `_classify_candidate_error`、`_is_auto_model`、`_auto_cooldown_seconds`、`_waf_blocked_suffix`、`_waf_blocked_hint` 作为委托入口，以维持既有调用兼容；这些入口不得再含有决策分支；
- 现有 `CandidateErrorClassifier` 如保留，必须降为 `ExecutionPolicyService` 的内部实现细节或薄转发层，不能与新服务形成两个策略 owner；
- 端口参数与 DTO 必须使用现有领域类型或明确 typing；禁止为便捷引入 `router: Any`、万能 `RouterPort` 或字典式分类结果作为新公共契约。

## 3. 硬冻结与禁止范围

以下执行路径仅可通过既有契约测试验证；不得改写其候选循环、fallback、状态写入时点、异常映射、response/iterator 生命周期或公开签名：

- `ArkProxyRouter.call`、`ArkProxyRouter.stream`、`ArkProxyRouter.finalize_stream_if_needed`；
- `CandidateRuntime.execute_non_stream`、`CandidateRuntime.execute_stream`；
- `NonStreamExecutionService.execute`、`StreamExecutionService.execute`；
- `RouterHandler.do_GET`、`RouterHandler.do_POST`、`create_server`。

禁止触碰：

- `/health`、`/api/*`、`/v1/*`、`/chat/*`、静态资源、`frontend/`；
- HTTP client、`UpstreamAdapter`、协议请求头/请求体构造、WAF 锁、日志/stream finalization；
- `ConfigStore` 持久化、配置 JSON 格式、模型/聚合排序、cooldown/success/unusable 写入语义；
- 外部错误 `type/code`、HTTP status、中文用户可见文案、日志字段与 fallback-chain 字段；
- 新增 retry、breaker、诊断、自动修复、配置迁移或任何业务能力。

## 5. 提交、回滚与 Gate

### 5.1 提交规则

I3 实施必须使用一个独立提交，提交消息建议为 `refactor: isolate execution policy port`。提交前必须确认：

```text
git diff --check
git diff --name-only a3265db..HEAD
git status --short
```

不允许将已有未提交的 G2 文档收口或未跟踪的 `frontend/` 混入 I3。若实施开始时工作区不干净，应只通过显式路径暂存 I3 文件；不得执行 `git add .`。

### 5.2 回滚基线

- **I3 代码回滚基线：** `a3265dbf848dade1cb1ce34ef052cc79245357a7`。
- **整体回滚方式：** I3 出现策略结果、fallback 行为、错误语义或导入边界回归时，完整回退 I3 独立提交至 `a3265db`；不得只回退端口或 service 的一部分，以免产生双 owner 或 facade 直连。
- **前置路由基线：** `/health` 可达性修复由 `b3e5ad3` 提供，I3 不得回退或改写该修复。

### 5.3 Gate

I3 Gate 必须同时具备：自动化结果、真实 `/health` smoke、diff/提交卫生、源码边界审计以及独立复核。任何一项缺失均不得宣布通过；只允许将未通过项记录为阻塞，不得以“无行为重构”豁免。

## 6. 后续边界

G3 只收敛执行策略端口。`ExecutionDependencies` 中请求构造、协议适配、并发、观测和 stream lifecycle 的窄端口迁移不属于 I3，须在 I3 Gate 通过后按一个领域一份冻结文档继续推进。
