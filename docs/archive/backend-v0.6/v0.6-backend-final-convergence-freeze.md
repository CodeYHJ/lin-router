# v0.6 后端最终收敛：执行依赖去宽桥冻结

- **状态：已冻结，允许一次最终集成实施**
- **Owner：架构-小强（边界/冻结）；全栈-小智（集中实施）；pm-小帅（最终 Gate）**
- **进入基线：** `9218abc`（G2/G3 文档收口）；代码基线为 I3 `38689bb`
- **整体回滚基线：** `9218abc`；若最终集成批次出现回归，必须整体回退该单一实现提交，不允许择一回退端口或 owner。
- **目标：** 在不改变 HTTP、配置、候选 fallback、流式生命周期和用户可见错误语义的前提下，消除执行面 `ExecutionDependencies` 宽桥，将其剩余实际消费收敛为有唯一 owner 的最小端口。
- **交付节奏：** 一次冻结 → 一次集中实现（可内部并行）→ 一次集成验证/独立复核 → 一次最终发布 Gate；本版本不再创建 G4/G5 子 Gate。

> 本文只冻结后端内部依赖方向，不等于“改写执行逻辑”。`frontend/` 是未跟踪独立工程，绝对不进入本批 diff、暂存或提交。

## 2. 最终模块边界与唯一 owner

| 最终领域/端口 | 唯一 owner | 允许能力 | 明确排除 |
|---|---|---|---|
| `CandidateQueryPort` + `HealthStatePort` | `CandidateHealthService` → `ConfigStore` | 连接组/聚合解析、候选枚举、成员 skip、success/unusable/cooldown 写入 | HTTP、锁、日志、策略决策 |
| `ExecutionPolicyPort` | `ExecutionPolicyService` | 错误分类、auto、冷却规则、WAF 文案 | Store、HTTP、锁、日志、候选缓存 |
| `RequestPreparationPort`（新增） | 新增纯 `RequestPreparationService`；协议细节复用 `UpstreamAdapter` | 路径解析、tools 顺序规范化、body 生成、鉴权/请求头生成、请求指纹所需的只读规范化结果 | 发送请求、Store 写入、锁、日志、HTTP Handler |
| `UpstreamRequestPort` | 现有 `UpstreamClient` | `request`、`close` | 业务决策、fallback、响应写回 |
| `ConcurrencyPort`（扩展为显式 lease） | `WafLockState` | 获得/释放候选 lease、流活跃计数、busy 事实 | HTTP、策略、配置/健康写入、日志持久化 |
| `StreamLifecyclePort`（新增） | 新增无状态 reader/usage adapter；资源归属仍由执行运行时 | idle timeout、逐行读取、response/stream usage 解析 | HTTP response 写出、候选决策、日志语义决定 |
| `RequestObservabilityPort` | `ObservabilityService` | live request、日志、stream lifecycle patch | fallback/健康/策略决策、上游调用 |
| `DebugCapturePort`（新增） | `DebugCapture` | 仅记录既有调试快照 | 参与请求发送、影响异常/返回值 |
| `ExecutionFaults`（新增不可变 DTO） | composition root / `runtime_types` | `AllModelsFailedError`、`StreamIdleTimeoutError`、`RouteContext` 等既有类型引用 | 任意 facade 或 `Any` 型 router |
| 兼容 facade / composition | `ArkProxyRouter` | 组装上述 owner；历史私有入口仅薄委托 | 再持有候选循环、策略、请求构造或 stream 状态机 |

### 2.1 目标依赖方向

```text
RouterHandler (HTTP transport, 不迁移)
  -> ArkProxyRouter compatibility facade (composition only)
  -> NonStreamExecutionService / StreamExecutionService
  -> CandidateRuntime coordinator
       -> CandidateQueryPort + HealthStatePort
       -> ExecutionPolicyPort
       -> RequestPreparationPort + UpstreamRequestPort
       -> ConcurrencyPort + StreamLifecyclePort
       -> RequestObservabilityPort + DebugCapturePort + ExecutionFaults

各端口实现 -> ConfigStore / UpstreamAdapter / UpstreamClient / WafLockState /
             ObservabilityService / DebugCapture
```

`CandidateRuntime` 保留“按既有顺序协调”的职责，但不得再保存或访问 `ExecutionDependencies`，不得反向 import `app`、`RouterHandler` 或 HTTP transport。`ExecutionDependencies` 在最终提交中必须删除；不允许保留“新端口 + 宽桥双路径”作为永久兼容层。

### 2.2 公共契约约束

- 新端口必须使用具名 DTO/明确类型；禁止新增 `router: Any`、`handler: Any`、`dict` 分类结果或万能 `RuntimePort`。
- `RequestPreparationPort` 的输出须为具名不可变 DTO（至少包含 target URL、outbound headers、body、body mode、tools-normalized 标记）；不得直接改变外部请求字段。
- `ConcurrencyPort` 返回具名 lease/上下文对象，统一封装 acquire/release、等待时长与释放原因；不得让执行循环读写 `WafLockState` 内部字典。
- `StreamLifecyclePort` 的 usage DTO 与现有五元组字段一一映射；不得新增或删除日志/fallback-chain 字段。
- `ExecutionFaults` 仅搬运现有类型，不改变抛出时点、捕获边界或 HTTP error 映射。

## 3. 集中实施顺序（一个提交，内部可并行）

| 工作流 | 前置 | 可并行性 | 产物/完成条件 |
|---|---|---|---|
| A. 契约与 DTO | 无 | 可与 B/C 并行 | 新端口、DTO、类型检查/导入边界测试；不接入运行时 |
| B. 请求准备与上游适配 | A 接口稳定 | 可与 C 并行 | `RequestPreparationService` + `UpstreamRequestPort` 装配，输出与基线请求字节/headers 等价 |
| C. 并发与流生命周期 | A 接口稳定 | 可与 B 并行 | lease、stream reader/usage adapter；锁释放、active-stream、idle-timeout 时点不变 |
| D. 观测与调试端口 | A 接口稳定 | 可与 B/C 并行 | 观测/DebugCapture 明确端口；日志字段、patch 时机与持久化格式不变 |
| E. Runtime 一次接线 | A-D 完成 | 串行，唯一集成点 | `CandidateRuntime` 改为只依赖窄端口，删除 `ExecutionDependencies`；双执行路径同次完成 |
| F. Facade 清理 | E 通过专项 | 串行 | `ArkProxyRouter` 只剩组装/历史薄委托；无反向 import、无第二 owner |

实施时允许多个开发流使用独立 worktree 产出 patch，但只能由一个集成分支合入，形成**单一、范围纯净提交**。禁止先合 A/B/C 的半成品提交再用“后续补齐”解释宽桥仍存在。

## 4. 硬冻结与禁止范围

以下行为必须与 `9218abc` 基线保持等价，实施中仅允许依赖注入/DTO 读取的机械变化：

1. `ArkProxyRouter.call`、`stream`、`finalize_stream_if_needed` 的公开签名、HTTP status/header/body、异常映射；
2. `CandidateRuntime.execute_non_stream`、`execute_stream` 的候选顺序、aggregate fallback、skip、cooldown/success/unusable 写入时点、请求级错误边界；
3. `RouterHandler.do_GET`、`do_POST`、`create_server`，以及 `/health`、`/api/*`、`/v1/*`、`/chat/*` 路由；
4. `ConfigStore` 格式、配置 JSON、模型/聚合排序、route key 与持久化语义；
5. `UpstreamAdapter` 的协议兼容规则、对外请求字段/headers、WAF 兼容策略；
6. 流式 chunk 转发、iterator close、finalize、日志字段、中文用户可见文案及 stable `type/code`；
7. `frontend/`、`static/`、打包配置、版本号、安装器及任何新功能（retry、breaker、诊断、自动修复、配置迁移）。

出现需要改动上述任一条的需求，必须从本批剥离为发布后独立需求；不得以“最终收敛”为名搭车。

## 6. 提交、回滚与最终 Gate

- **实施提交：** 一次集中实现，建议消息 `refactor: finalize execution dependency ports`；仅显式暂存允许文件，严禁 `git add .`。
- **文档收口：** 最终 Gate 证据以单独 docs-only 提交归档；业务代码回滚基线与文档提交必须分别记录。
- **回滚：** 若任一协议、fallback、stream lifecycle、锁或观测语义回归，整体回退最终实施提交至 `9218abc`；不得局部撤回单个 port。
- **最终 PASS 条件：** 宽桥已删除、唯一 owner/端口已实证、全部验证矩阵通过、独立复核 PASS、安装包首次运行通过。缺任一项，v0.6 只能称“代码收敛完成，发布 Gate 未通过”。

## 7. 风险与取舍

- **收益：** 以一次集成 Gate 替代 G4/G5 多轮等待，完成执行面剩余所有依赖收敛，后续新增能力可落在明确 owner 上。
- **代价：** 单提交影响面较大，定位成本高于微批次；因此必须在 A-D 先以契约和独立 patch 对齐，再由 E 单点集成。
- **最高风险：** 非流/流两套近似循环容易出现一侧接线遗漏；WAF lease 和 stream finalize 的资源释放顺序不能靠重构直觉修改。
- **缓解：** 新增“同一端口两路径均消费”的静态断言；对 `call/stream`、handler、server 做基线 AST 对比；真实 upstream + 安装包首次运行双层验证。
