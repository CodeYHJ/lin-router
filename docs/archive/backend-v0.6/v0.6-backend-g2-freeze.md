## 1. 唯一抽离边界与状态 owner

新增 `CandidateHealthService`（等价命名可接受）作为候选查询、聚合成员跳过判定及健康状态写入的唯一业务 owner。

| 领域 | 唯一 owner / 协作者 | I2 结论 |
|---|---|---|
| groups、models、aggregate_members 与持久化 | `ConfigStore` | 继续为唯一数据 owner；服务不得保留第二副本 |
| 候选构造、枚举、聚合成员 skip、成功/cooldown 写入 | `CandidateHealthService` | 从 `CandidateRuntime` 迁出 |
| WAF 锁 | `WafLockState` | 不迁移 |
| 观测与日志 | `ObservabilityService` | 不迁移 |
| 执行协调 | 既有执行服务 / `CandidateRuntime` facade | 保持调用兼容，仅消费新服务能力 |

迁移的既有职责仅包括：

- `iter_candidates`、`iter_upstream_candidates`、`iter_aggregate_candidates`；
- `aggregate_member_skip_reason`、`candidate_from_model`；
- `set_cooldown`、`set_success`。

## 2. 端口、依赖与兼容约束

1. 仅可细化既有 `CandidateQueryPort`、`HealthStatePort`；使用明确 DTO：`UpstreamCandidate`、模型/成员标识与健康写入参数。
2. 禁止新增 `router: Any`、万能 `RouterPort`、反向 `import app`；新服务、端口与 DTO 不得依赖 `RouterHandler` 或 HTTP transport。
3. `CandidateRuntime` 可保留为薄 facade / 组合器，保护现有执行调用路径；不得形成双 owner。
4. `ArkProxyRouter` 的公开入口、构造签名与外部可见语义保持不变。

依赖方向：

```text
执行服务 / CandidateRuntime compatibility facade
  -> CandidateHealthService
  -> CandidateQueryPort + HealthStatePort
  -> ConfigStore
```

## 3. 硬冻结与禁止范围

以下方法只允许验证，不得修改业务分支、参数/返回元组、异常映射或流生命周期：

- `ArkProxyRouter.call`、`ArkProxyRouter.stream`；
- `CandidateRuntime.execute_non_stream`、`CandidateRuntime.execute_stream`；
- `RouterHandler.do_GET/do_POST`、`create_server`。

禁止触碰：

- `/health`、`/api/*`、`/v1/*`、静态资源、`frontend/`；
- HTTP client、`UpstreamAdapter`、WAF 锁、日志/流 finalization、普通/流式 fallback；
- 配置 JSON 格式、错误 `type/code` 与用户可见文案；
- breaker、诊断、自动修复或其他新能力。
